# 协议分析

## 信号特征

| 参数 | 值 |
|------|-----|
| 捕获方法 | ESP32 MicroPython 边沿轮询（上升沿+下降沿交替） |
| 采样精度 | 微秒级（`time.ticks_us()`） |
| 物理介质 | 单线直连（无104电容）|
| B线空闲电压 | 2.32V |
| ESP32拉低幅度 | 2.32V → 0.43V（Δ=1.89V） |

## 编码体系

### 三态脉冲宽度编码

| 符号 | 宽度范围 | 物理含义 |
|------|----------|----------|
| **0** | ~485μs (<700μs) | 数据位0 |
| **1** | ~975μs (800-1100μs) | 数据位1 |
| **E** | ~1250-1600μs | 第三态 / 按键矩阵行扫描标记 |
| **F** | >4000μs | 帧心跳分隔符 |

### 帧结构

- **帧格式**：可变长 3-14 bit
- **帧间隔**：F 脉冲分隔（>4000μs）
- **帧密度**：空闲~2.4fps，按键时~50fps
- **稳定帧**：`0000001000010`（送风低风特征帧）×57 (31%)

## 🔥 核心协议模型

### 信号1的密度 = 压缩机需求度

**面板不广播"目标温度=17"这个数字，而是广播"压缩机需求强度"**。

1 的密度越高 → 需求越大 → 压缩机转速越高（或风扇档位越高）。

| 状态 | 1 占比 | 含义 |
|------|--------|------|
| 制热 17°C 低风（室温远高） | 0.0% | 需求=0 |
| 送风 17°C 低风 | 12.7% | 风扇低速 |
| 送风 17°C 高风 | 22.8% | 风扇高速 |
| 制冷 24°C 低需求（接近室温） | 22.1% | 接近平衡 |
| 制冷 17°C 高需求（远离室温） | 45.4% | 需求最大 |

**5状态完全验证，模型自洽**。

### 风扇独立编码

送风模式下的 `0000001000010` 是稳定特征帧（占31%），与压缩机的 0/1 密度信号并行。

### 温度与需求的对应

- 制冷 17°C → 室温(30°C) 差 13°C → 需求大 → 1 占比 45%
- 制冷 24°C → 室温(30°C) 差 6°C → 需求中 → 1 占比 22%
- 制热 17°C → 室温(30°C) 差 -13°C → 需求=0 → 1 占比 0%
- 送风 → 不涉及压缩机 → 1 占比仅反映风扇档位

## 协议模型演进

### 模型1：按钮命令编码 ❌（06-21~06-27上午）
假设"按模式键→面板发送模式码"。推翻：所有按钮帧长分布几乎相同。

### 模型2：特定码=特定按钮 ❌（06-27晚）
假设`0011011`=模式键、`0000000`=温度+。推翻：交叉验证发现两个码出现在多个按钮甚至idle中。

### 模型3：完整状态数据包 🔄
面板每次广播所有状态参数快照。有道理但无法确定字段边界。

### 模型4：二维符号编码 ✅（06-27~28）
(HIGH宽度, LOW间隔) 二维符号，共42种可能，实际31种。共同符号集。

### 模型5：三态7-bit帧 🔄（06-28起）
三态编码确立。TEMP+ 剥E后全零。

### 模型6：全零=无负载 ✅（07-04 修正）
修正：TEMP+ 全零不是因为"最高温"，而是因为当时在制热+室温已达，主机停转。

### 模型7：需求度模型 ✅（07-05 验证）
**最终模型**：信号1的密度 = 压缩机需求强度。5状态对比全验证通过。

## 关键发现

1. **协议不是数值编码**——面板不广播"目标温度=17"
2. **协议是需求度模型**——1的密度 = 压缩机需求强度
3. **风扇独立编码**——送风模式用 `0000001000010` 稳定帧
4. **总线上双向通信**——面板和主机都在说话，回放必须隔离
5. **ESP32 能直接驱动B线**——2.32V 拉到 0.43V，回放物理层可行
6. **bit 字段独立语义未解**——bit0-6 各对应什么参数未分离，但需求度模型已能控制压缩机
