# 进度与计划 — 世纪星空调协议逆向

## 当前进度

### ✅ 已完成（2026-07-05 协议解码完成）

**物理层**
- ESP32 D33 ← B线（直接连，无104电容），GND ← A线
- 物理层验证：ESP32 3.3V 可拉低B线（2.32V → 0.43V，Δ=1.89V）
- 通信方式已排除：红外、RS485、模拟电压、低压串口直读、100kΩ上拉
- Mac ESP32 CH340 驱动问题已修复

**协议解码**
- **编码方式**：三态脉冲宽度（0≈485μs / 1≈975μs / E≈1250-1600μs / F>4000μs）
- **帧边界**：F脉冲（>4000μs）分隔，E脉冲（1250-1600μs）参与帧内编码
- **协议模型**：面板持续广播状态寄存器快照，非"按一次发一个命令"
- **协议含义**：**信号1的密度 = 压缩机需求度**（非"目标温度"数值编码）

**协议模型确认（07-05 五状态全验证）**：

| 状态 | 1占比 | 含义 | 验证状态 |
|------|-------|------|---------|
| 制热 17°C 低风（室温远高于17°C） | **0.0%** | 需求=0，压缩机停 | ✅ |
| 送风 17°C 低风 | 12.7% | 风扇低速 | ✅ |
| 送风 17°C 高风 | 22.8% | 风扇高速 | ✅ |
| 制冷 24°C 低需求（接近室温） | 22.1% | 接近平衡 | ✅ |
| 制冷 17°C 高需求（远离室温） | **45.4%** | 需求=1，压缩机全速 | ✅ |

**五状态帧 Top1 差异**：
- 制热17°C低风：`0000000000000` (13个全0) ×49 → 纯负载=0
- 送风17°C低风：`0000001000010` ×57 → 风扇低特征
- 送风17°C高风：`100` ×11 → 风扇高特征
- 制冷17°C高需求：`100` ×31 + `110` ×10 → 压缩机高需求
- 制冷24°C低需求：`000` ×16 + `0000101001010` ×14 → 接近平衡

### 🔄 进行中：回放验证

**状态**：三种纯软件方案全部试过，全部失败。

**三次尝试（2026-07-08）**：

| 方案 | 方法 | 结果 | 失败原因 |
|------|------|------|----------|
| RMT 硬件脉冲 | ESP32 RMT 外设生成精确时序 | ❌ 无反应 | 总线竞争 |
| 双 GPIO 伪差分 | D33→B(信号), D26→A(反相) | ❌ 无反应 | 总线竞争 + A线通过104电容到GND无法独立驱动 |
| 开漏驱动 | D33 只拉低不推高 | ❌ 无反应 | 总线竞争 |

**根因确认**：面板始终在发信号（~400ms 一帧），ESP32 发帧时两个信号在 B 线上叠加碰撞。面板不下线，ESP32 永远抢不到总线。纯软件方案已穷尽。

**解决方向（硬件）**：
- 5V 单路继电器模块（已购买，等快递）
- 原理：回放时继电器吸合 → 面板从 B 线断开 → ESP32 独占总线
- 接法：B线剪断 → COM 接主机端, NC 接面板端, NO 接 ESP32 D33
- ESP32 D25 控 IN 端；Low 触发（吸合=回放, 释放=面板正常）

## 关键发现

1. **协议不是数值编码**——面板不广播"目标温度=17"
2. **协议是需求度模型**——1的密度 = 压缩机需求强度
3. **风扇独立编码**——送风模式用 `0000001000010` 稳定帧
4. **总线上双向通信**——面板和主机都在说话，回放必须隔离
5. **ESP32 能直接驱动B线**——2.32V 拉到 0.43V，回放物理层可行
