# 进度与计划 — 世纪星空调协议逆向

## 当前进度

### ✅ 已完成的

**硬件**
- ESP32 D33 ← B线（104电容耦合），GND ← A线。接线稳定可靠
- 通信方式已排除：红外、RS485、模拟电压、低压串口直读、100kΩ上拉
- Mac ESP32 CH340 驱动问题已修复（SIP关闭后kext加载成功）

**协议解码**
- **物理层**：三态脉冲宽度编码（0≈485μs / 1≈975μs / E≈1250-1600μs / F>4000μs）
- **帧结构**：7-bit，F脉冲间隔~395ms（空闲），按键时帧密度暴增
- **编码模型**：面板持续广播状态寄存器值（非"按一次发一个命令"）
- **TEMP+ 为最简态**：31帧，剥E后全零，是基准态
- 六按钮 idle/mode/temp+/temp-/wind/power 已各采集15秒数据

**工具**
- `tools/esp_capture.py`：ESP32边沿捕获（MicroPython）
- `tools/analyze.py`：三态脉冲分类+7-bit帧分析+二维符号统计+跨文件对比
- `tools/capture_diff.py`：差分采集→分析一键脚本
- `tools/cap_and_analyze.py` + `空调捕获.bat`：Windows采集入口

### 🔄 进行中：温度编码系统推导

**2026-07-04 重大发现**（TEMP+/TEMP-/IDLE 比特流对比分析）：

**E脉冲 = 3行按键矩阵扫描标记**：
- TEMP+ 帧中 E 在 pos0/2/3/5 轮转，稳定帧为 `0000000` → 所有数据位 = 0
- 剥E后数据位全零 → **TEMP+ = 状态寄存器全零** = 最高温基准态

**温度编码模型**（证实但未完成映射）：
- TEMP+（最高温）= 217位全零 → 18位状态字全零
- TEMP-（较低温）= 321位有1 → 单1在6个数据位轮转
- 温度越低 → 1越多 → 编码可能是"温度计数"（非二进制）
- **已确认**：`0011000` 和 `10011` 是中间温度码，不是固定的20/17°C

**IDLE 模式特征**：
- 273位比特流，无重复周期
- 每帧不同，面板在持续广播完整状态

## 关键发现

| 发现 | 含义 |
|------|------|
| TEMP+全零 | 最高温=18位状态字全零（基准态） |
| E脉冲轮转 | 3行按键矩阵扫描标记（pos0/3/5），非数据位 |
| 温度编码 | 温度越低1越多，可能是温度计数/step量=编码值 |
| 按键vs稳定 | 按键时有E扫描干扰，稳定idle时才发纯数据帧 |
| F脉冲不可靠 | 帧边界F脉冲有时不出现，导致帧长飘动 |
