# 进度与计划 — 世纪星空调协议逆向

## 当前进度

### ✅ 已完成的

**硬件**
- ESP32 D33 ← B线（104电容耦合），GND ← A线。接线稳定可靠
- 通信方式已排除：红外、RS485、模拟电压、低压串口直读、100kΩ上拉
- Mac ESP32 CH340 驱动问题已修复（SIP关闭后kext加载成功）

**协议解码**
- **物理层**：三态脉冲宽度编码（0≈485μs / 1≈975μs / E≈1250-1600μs / F>4000μs）
- **帧结构**：7-bit，F脉冲间隔~395ms（空闲），按键时帧密度暴增
- **编码模型**：面板持续广播状态寄存器值（非"按一次发一个命令"）
- **TEMP+ 为最简态**：31帧，剥E后全零，是基准态
- 六按钮 idle/mode/temp+/temp-/wind/power 已各采集15秒数据

**工具**
- `tools/esp_capture.py`：ESP32边沿捕获（MicroPython）
- `tools/analyze.py`：三态脉冲分类+7-bit帧分析+二维符号统计+跨文件对比
- `tools/capture_diff.py`：差分采集→分析一键脚本
- `tools/cap_and_analyze.py` + `空调捕获.bat`：Windows采集入口

### 🔄 进行中：温度字段差分

**2026-07-04 捕获成果**：
- **20°C = `0011000`**（7位） ✅ 已确认（MODE采集松手帧 `00110011` 的截断）
- **17°C = `10011`**（5位） ✅ 已确认（单次TEMP-松手帧，3次独立采集验证）
- 18°C ≈ `000000`（6位） ⚠ 待确认
- 19°C ≈ `0000`（4位） ⚠ 待确认

**核心发现**：
- 帧长随温度变化：20°C(7位) → 19°C(4位) → 18°C(6位) → 17°C(5位)
- 温度编码**不是**标准二进制/格雷码，帧长浮动暗示可能是变长编码
- 松手瞬间帧 vs 按键中短帧需要区分（按键中是按键扫描帧，松手才是状态帧）

### ❌ 未完成的

**最核心的：温度编码规律**
- 7位→5位的变化规律未确定
- 温度字段bit位位置待定位
- 需补齐 16°C / 15°C / ... 等更多温度点的编码

## 关键发现

| 发现 | 含义 |
|------|------|
| TEMP+全零 | 温度最高时所有数据位=0（某参数达到极值的编码方式） |
| IDLE的bit5=46% | idle状态下此位频繁翻转，可能是运行/待机状态位 |
| MODE的bit1=57% | 模式切换时此位变化最剧烈 |
| 帧间翻转率普遍40-60% | 每帧都不同 → 面板在持续广播整个状态寄存器，非增量更新 |
